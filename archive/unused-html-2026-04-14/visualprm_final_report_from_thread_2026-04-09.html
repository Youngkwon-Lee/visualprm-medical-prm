3) 상세 설명 드롭다운 (요청 반영)

에이전트(specialist)별 역할은 무엇인가?
현재 policy 기준 주요 specialist:
  • general_medical_reasoner: 기본 fallback reasoning
  • radiology_reasoner: 영상의학 질문(CT/MRI/X-ray/uptake/pattern 등)
  • pathology_reasoner: 병리 질문(nucleus/h&e/histology/atypia 등)
  • modality_recognition_reasoner: 영상 modality 인식형 질문
분기 입력: dataset, case_type, modality, question_type, 질문 키워드.
RAG는 어떻게 만들어졌나?
  • 케이스 텍스트(데이터셋/질문/옵션/모달리티 등)를 벡터화
  • 기본 임베딩: hash-fallback (외부 API 없이 동작)
  • 옵션 임베딩: sentence-transformers (설정시)
  • 질문이 오면 유사 케이스 검색 + reranker 정렬 + document 근거 결합
qdrant는 어떤 워크플로우로 동작하나?
  1. 서버 시작 시(또는 최초 호출 시) qdrant 컬렉션 생성: medical_vqa_cases
  2. 데이터셋 케이스를 포인트로 upsert (payload: dataset/id/question/options/modality/image_url)
  3. 질문마다 query vector 생성 후 top-k 검색
  4. allowed_datasets, exclude_same_case 필터 적용
  5. 검색 결과를 rerank 후 최종 reasoning에 투입
현재 운영 모드: RAG_QDRANT_MODE=memory, RAG_AUTO_WARM=1 기준 사용.
우리 데이터셋 예시로 설명 (실제 실행 케이스)
예시 케이스: pathvqa_000000
질문: Is the nucleus enlarged?
  1. 입력: pathvqa + 이미지 + options(Yes/No)
  2. Router: pathology_reasoner로 분기
  3. Retrieval: qdrant에서 유사 케이스 검색
  4. Document RAG: 병리 문서 근거 결합
    • Nuclear enlargement
    • Hyperchromasia and atypia
  5. 출력: /agent-answer 200 + final answer + document_hits 반환
VLM 생성 경로 최신 상태 (복구 후 결과)
  • Gemini API 키/한도 갱신 후 /generate-steps 재실행 성공
  • PathVQA 50샘플 평가 완료: 34/50 (68%)
  • 평균 지연: 13.09s
  • 비-200 응답 9건은 대부분 JSON 포맷 이탈(파싱 오류) 케이스
  • 비교: /agent-answer는 26/50 (52%), 평균 0.18s
결론: Gemini VLM 생성은 다시 동작하며, 정확도는 높지만 응답시간/출력 안정화(JSON 강제)가 추가 과제.
RunPod 시도 결과 요약
  • A5000/4090/3090 + secure/community 다중 시도
  • pod 생성/RUNNING은 가능
  • SSH pub port 미할당 또는 connection refused 반복
  • 모든 테스트 pod stop/remove로 비용 정리 완료
