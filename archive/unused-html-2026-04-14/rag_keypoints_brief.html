RAG + Agent 활용

핵심 구조

Route질문의 도메인과 유형을 먼저 나눔
→
Retrieve관련 자료를 넓게 검색
→
Rerank더 믿을 만한 근거만 다시 고름
→
Select필요한 만큼만 context 선택
→
Generate근거를 읽고 모델이 답함
의료 이미지 예시: 답 도출용 RAG
예를 들어 PathVQA / H&E / “Is the nucleus enlarged?” / Yes-No 문제라면, 먼저 pathology 문제로 분류하고, 비슷한 병리 사례를 검색한 뒤, 그중 더 관련 높은 사례만 다시 고르고, 필요한 근거만 남겨서 reasoner가 최종 답을 만듭니다.
의료 이미지 예시: 풀이과정 검증용 RAG
같은 문제에서 모델이 “핵이 커 보이므로 Yes”라고 중간 step을 만들었다면, RAG로 병리 설명 문서나 가이드라인, 유사 근거를 가져와서 이 step이 실제 근거와 맞는지 확인하는 방식입니다. 즉 이 경우 retrieval은 답을 만드는 데만 쓰이는 게 아니라, reasoning 과정이 타당한지 검증하는 데도 쓰입니다.

논문 사례

ICLR 2025

MMed-RAG

분야별로 검색기를 나누고, similarity에 따라 필요한 근거만 고릅니다. domain-aware retrieval과 adaptive context selection을 함께 쓰는 사례입니다.

Information Fusion 2025

FGRW

PEIR, ROCO 같은 image-text knowledge를 검색한 뒤, 그중 어떤 근거를 더 중요하게 볼지 다시 조정해 답변 성능을 높입니다.

Neurocomputing 2025

MKGF

문서 chunk만 쓰지 않고, multi-modal medical knowledge graph를 retrieval source로 씁니다. graph, entity, image 정보를 함께 쓰는 방향입니다.

2025

Med-PRM

retrieval을 최종 답 생성보다 intermediate reasoning step 검증에 더 가깝게 씁니다. 가이드라인과 문헌으로 중간 풀이 과정이 맞는지 확인합니다.

우리 현재 방향

  • 문제를 먼저 분류합니다.
  • Qdrant로 비슷한 사례를 찾습니다.
  • 검색 결과를 다시 정렬합니다.
  • 필요한 근거만 추려서 씁니다.
  • 과정을 UI에서 보여줍니다.
쉽게 말하면
우리도 최신 논문들과 비슷하게, 먼저 고르고, 다시 정렬하고, 그 근거를 바탕으로 답하게 만드는 흐름을 따라가고 있습니다.

오픈소스로 바로 가능한 스택

Agent

LangGraph

route → retrieve → rerank → reason 같은 agent 흐름을 그래프로 관리하기 좋습니다.

RAG

Haystack / LlamaIndex

retriever, reranker, document ingestion, evaluation까지 빠르게 붙이기 좋습니다.

Vector DB

Qdrant / pgvector

Qdrant는 빠른 실험용, pgvector는 기존 Postgres와 함께 쓰기 좋습니다.

Embedding

BGE / SentenceTransformers / BiomedCLIP

텍스트는 BGE 계열, 이미지나 의료 멀티모달은 BiomedCLIP 같은 후보를 고려할 수 있습니다.
